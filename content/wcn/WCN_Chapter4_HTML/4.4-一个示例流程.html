4.4 · 示例流程

用一个真实风格的案例，把 WCN 的最小闭环完整走一遍。

抽象流程看懂了，但系统真正有说服力的时候，是你能把它放进具体场景里。这里用一个“项目融资 + 服务协同”的复合闭环示例，把整条路径跑一遍。

本页作用
把抽象逻辑落到现实案例
核心主题
节点协作、Agent 执行、Proof 与 PoB
阅读重点
全过程、证据与归因

案例角色

项目节点一个正在融资的 AI + RWA 项目。
资本节点一家对 AI 与资产结构有兴趣的基金。
服务节点提供法务与结构支持。
Research Agent负责整理材料与初步尽调。
Setup

场景设定

一个项目节点进入 WCN，希望完成两件事：

完成种子轮融资需要找到合适的资本方，并提高被理解和被推进的效率。
明确法务与资产结构需要服务节点帮助整理项目结构与材料。

与此同时，WCN 网络中已经有：

资本节点
偏好 AI / RWA，票面与阶段都匹配。
服务节点
可以提供法务与交易结构支持。
Research Agent
负责对项目资料做摘要、风险整理与初筛。
Deal Owner
负责把整个流程从引荐推进到结果。
Steps

步骤 1 到 6

步骤 1：项目节点进入系统

项目方提交资料、当前阶段、融资需求、目标地区与重点问题，进入 Project Intake。

步骤 2：Deal Room 被创建

系统为该项目创建一个 Deal Room，明确 owner、潜在资本节点、服务节点与下一步动作。

步骤 3：Research Agent 先做材料整理

Agent 对项目资料做摘要、补充缺口清单、列出初步风险与投资判断要点，并把结果回写到 Deal Room。

步骤 4：资本节点与服务节点被拉入

资本节点收到结构化资料，服务节点收到法务与结构任务。两条线同时推进，而不是串行等待。

步骤 5：会议、尽调与补件推进

Deal Owner 组织会议，Execution Agent 记录纪要、追踪待办、催促补件，Research Agent 继续更新判断。

步骤 6：形成结果并进入 Proof

一旦融资达成、协议签署或服务交付通过验收，系统把对应结果推进到 Proof Desk。

WCN 的价值不在于“让大家碰面”，而在于让每一步都能留下 owner、状态、输出和证据。
Proof

证据与归因怎么做？

这个案例里，至少会形成三类证据：

融资证据
投资意向、协议、打款、确认邮件、链上记录等。
服务证据
法务交付件、结构 memo、SOW 验收记录。
Agent 证据
研究摘要、版本记录、输入来源、会议纪要、催办日志。
流程证据
谁引荐、谁推进、谁组织、谁审核、谁采纳，都有状态和日志。

随后，系统再做归因：

主导节点负责主要推进并对结果承担责任的人。
协作节点提供关键辅助但不构成主导的人。
Agent 贡献只有在输出被采纳并进入闭环时才有价值。
审核与结论通过 Proof Desk 后，结果进入 PoB 账本。
Result

最后系统得到什么？

项目方得到结果
融资和服务问题同时推进，降低了协作成本。
资本方得到更清晰判断
信息结构化、沟通更高效、DD 成本更低。
服务方进入正式闭环
服务交付与后续结算都更清楚。
网络得到 PoB 输入
这次闭环不会消失，而会沉淀为长期网络价值。
这就是 WCN 和普通介绍群、CRM 或单点工具的区别：它让一次协作，不只做成结果，还变成系统资产。
