## 5 つの代表的なエージェント

<CardGrid cols={2}>
  <Card title="研究エージェント" icon="search">情報の検索、複数の情報源の概要、比較可能な企業/トラックの構造化された表、およびデューデリジェンスリストの記入。 **ツールの例**: イントラネット ドキュメント RAG、許可されたサードパーティ データ API、Web ページ クローリングへの直接接続を禁止します (アンチインジェクション)。 **LLM 注**: ファクト参照にはソース アンカーが必要です。値はデフォルトで「検証対象」としてマークされます。</Card>
  <Card title="ディールエージェント" icon="handshake">資本プロジェクトのマッチ候補、ティーザーレベルのピッチドラフト、取引段階のステータスの推奨事項。 **禁止**: 承認なしにノードに代わって拘束力のある条件または排他的なコミットメントを発行すること。 **Copilot の比較**: Copilot は 1 人による原稿を最適化します。取引エージェントの出力には、レビュー用の deal_id と推奨ロジックの概要が必要です。</Card>
  <Card title="成長剤" icon="trending-up">マルチバージョンのコピー、ファネルの箇条書き、キャンペーン後のアトリビューションの概要 (成長タスクに関連付けられている)。 **リスク**: 外部に公開する前に手動リリース ゲートを通過する必要があります。 「利益コミットメント」などの即効性のある言葉と組み合わせるのは避けてください。</Card>
  <Card title="執行代理人" icon="settings">会議議事録、To Do の内訳、リマインダーの下書き、領収書追跡リスト。 **ベンチマーク**: エンタープライズ自動化 (Zapier/Make) + LLM に最も近いですが、各提案は会議メモとタイムスタンプにリンクされている必要があります。</Card>
  <Card title="流動性エージェント" icon="droplet">流動性と市場の健全性モニタリングの概要、チェックリストのステータス、異常なしきい値アラームの説明。 **TradFi Lenovo**: Aladdin の特定タイプのリスク ダッシュボードの「自然言語層」と同様に、意思決定権限は依然としてトレーディング デスク/ノードにあります。</Card>
</CardGrid>

## なぜ「スーパーエージェント」に混ぜてはいけないのでしょうか？

LangChain スタイルの汎用エージェントは、多くの場合、同じツールセットを共有します。組織的なシナリオでは、**最小特権の原則**に違反することになります。分類後、システムは次のことを実行できます。

<CardGrid cols={2}>
  <Card title="誰が何を見れるのか" icon="eye">研究では幅広い資料を読むことができます。流動性は市場価格と合意された契約のみを読み取ることができます。 Deal は、デフォルトではデータベース全体の給与レベルの機密フィールドを読み取りません。</Card>
  <Card title="誰が何をできるのか" icon="wrench">実行ではタスクとコメントを書き込むことができます。 Growth はマーケティング ドラフト領域のみを作成できます。どのタイプのエージェントも、デフォルトでは送金したり契約に署名したりすることはできません。</Card>
  <Card title="誰がどのログを保管すべきでしょうか?" icon="clipboard-list">Research は検索クエリとフラグメント ID を記録します。 Deal は推奨される機能とバージョンを記録します。流動性はインジケーターの定義とデータ時間を記録します。</Card>
  <Card title="誰がどのようにして PoB に参加するのか" icon="circle-check">帰属テンプレートはタイプごとに分割されています。たとえば、「採用された研究の段落」と「採用された成長資料」には、和解紛争を避けるために異なる証明フィールドがあります。</Card>
</CardGrid>

## マルチエージェントオーケストレーションによるマッピング関係

CrewAI / LangGraph の *エージェント / ノード* は WCN のタイプと比較できますが、WCN ではさらに 3 つの点が修正されています。 **(1)** 各タイプで許可されるツールのホワイトリスト。 **(2)** 人間の承認エッジを持つステート マシン。 **(3)** 出力は `task_id`、ノード採用イベントにバインドされます。 (3)がなければ、どんなに美しい配置でも、それはただの内部実験に過ぎません。

<Callout type="warning">
機密化されていない → 権限の拡張 → ログを防御できない → PoB の価格設定ができない。タイプはガバナンスと商業化の前提条件であり、文書の装飾ではありません。
</Callout>

## 短い例: 同じイベント、複数種類のコラボレーション

<Steps>
  <Step title="研究">公的融資ニュース＋社内メモを引っ張り出し、「対立点と疑問点リスト」を生成。</Step>
  <Step title="取引">リストに基づいて、一致する理由を付けて「次回のコミュニケーション ポイント」のドラフトを出力します (自動メール送信はありません)。</Step>
  <Step title="実行">会議の結論を所有者/期限付きのアクションアイテムとして書き込み、タスクに添付します。</Step>
  <Step title="成長（オプション）">採用されたファクトから外部の中立的な更新コピーを生成し、手動でクリックして公開します。</Step>
</Steps>

<Statement>タイプの目的は、アーキテクチャ レベルで回答することです。何か問題が発生した場合、それはモデル、ツール、構成、または人間の採用に問題があるのか​​、管理する前に分解することができます。</Statement>
