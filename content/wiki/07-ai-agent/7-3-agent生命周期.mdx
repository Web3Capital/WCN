## Agent 的关键状态

<CardGrid cols={3}>
  <Card title="Registered" icon="📝">Agent 已被登记，但还没进入正式业务流程。</Card>
  <Card title="Authorized" icon="🔑">Agent 被赋予明确任务、数据和权限边界。</Card>
  <Card title="Active" icon="▶️">Agent 正在执行任务并持续写入日志。</Card>
  <Card title="Paused / Suspended" icon="⏸️">因质量、越权或风险问题暂停。</Card>
  <Card title="Upgraded" icon="⬆️">Agent 升级模型能力或权限等级。</Card>
  <Card title="Retired" icon="🏁">Agent 因不再适用或不达标而退出系统。</Card>
</CardGrid>

## 生命周期路径

<Callout type="info">
注册 → 授权 → 上线 → 日志记录 → 审核 / 复核 → 升级 / 暂停 / 淘汰
</Callout>

<Statement>生命周期的设计目标不是增加复杂度，而是保证每个 Agent 可被识别、可被约束、可被淘汰。</Statement>

---

## 生命周期保证什么

<CardGrid cols={2}>
  <Card title="每个 Agent 都有身份" icon="🪪">知道是谁、做什么、属于谁。</Card>
  <Card title="每个 Agent 都有边界" icon="🛡️">不能无约束读取和执行。</Card>
  <Card title="每个 Agent 都有日志" icon="📋">可以回溯与审查。</Card>
  <Card title="每个 Agent 都能被淘汰" icon="❌">质量差或越权的 Agent 不能无限存在。</Card>
</CardGrid>

<Callout type="warning">
没有生命周期，Agent 既会失控（无授权与暂停机制），也无法进入 Proof（无身份和日志）。
</Callout>
