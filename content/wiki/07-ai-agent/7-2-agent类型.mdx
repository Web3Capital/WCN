## 五种典型 Agent

<CardGrid cols={2}>
  <Card title="Research Agent">信息检索、多源摘要、可比公司/赛道结构化表、尽调清单填充。**工具示例**：内网文档 RAG、许可的第三方数据 API、禁止直连任意网页抓取（防注入）。**LLM 注意**：事实引用必须带来源锚点；数值默认标「待核对」。</Card>
  <Card title="Deal Agent">资本—项目匹配候选、Teaser 级介绍语草稿、deal 阶段状态建议。**禁止**：擅自代表节点发具约束力条款或排他承诺。**Copilot 对比**：Copilot 优化单人文稿；Deal Agent 输出需带 deal_id 与推荐逻辑摘要供复核。</Card>
  <Card title="Growth Agent">多版本文案、渠道要点、活动后归因摘要（与 Growth 任务绑定）。**风险**：对外发布前必经人工发布闸；避免与「收益承诺」类提示词组合。</Card>
  <Card title="Execution Agent">会议纪要、待办拆解、催办话术草稿、回执追踪列表。**对标**：最接近企业自动化（Zapier/Make）+ LLM，但每条建议应可链接到会议记录与时间戳。</Card>
  <Card title="Liquidity Agent">流动性与盘口健康度监控摘要、上所 checklist 状态、异常阈值告警说明。**TradFi 联想**：类似 Aladdin 中某类风险看板的「自然语言层」，决策权仍在交易台/节点。</Card>
</CardGrid>

## 为何不能混成一个「超级 Agent」

LangChain 式通用 Agent 往往共享同一工具集；在机构场景会导致**最小权限原则**被破坏。分类后系统可以做到：

<CardGrid cols={2}>
  <Card title="谁能看什么">Research 可读广泛材料；Liquidity 仅读行情与约定合约；Deal 默认不读全库薪酬级敏感字段。</Card>
  <Card title="谁能做什么">Execution 可写任务与评论；Growth 仅写营销草稿区；无一类 Agent 默认可转账或签约。</Card>
  <Card title="谁该留什么日志">Research 记检索查询与片段 ID；Deal 记推荐特征与版本；Liquidity 记指标定义与数据时间。</Card>
  <Card title="谁如何进入 PoB">归因模板按类型拆分：例如「采纳的研究段落」与「采纳的增长素材」走不同 Proof 字段，避免结算扯皮。</Card>
</CardGrid>

## 与多智能体编排的映射关系

CrewAI / LangGraph 中的 *agent / node* 可类比 WCN 的类型，但 WCN 额外固定三件事：**（1）** 每类型允许的工具白名单；**（2）** 与人类审批边的状态机；**（3）** 输出与 `task_id`、节点采纳事件的绑定。没有（3），再漂亮的编排也只是内部实验。

<Callout type="warning">
不分类 → 权限膨胀 → 日志无法辩护 → PoB 无法计价。类型是治理与商业化的前置条件，不是文档装饰。
</Callout>

## 短例：同一事件，多类型协作

<Steps>
  <Step title="Research">拉取公开融资新闻 + 内部 memo，生成「冲突点与待问清单」。</Step>
  <Step title="Deal">基于清单输出「下一轮沟通要点」草稿，附带匹配理由（不自动发邮件）。</Step>
  <Step title="Execution">把会议结论写成带 owner/due 的行动项并挂到 Task。</Step>
  <Step title="Growth（可选）">从已采纳事实生成对外的中性更新文案，人工点击发布。</Step>
</Steps>

<Statement>类型的目标是**在架构层回答**：若出事，是模型、工具、配置还是人类采纳环节的问题——可拆分才可治理。</Statement>
