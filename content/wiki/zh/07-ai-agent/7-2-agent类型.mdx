## 五个 MVP Agent

<CardGrid cols={2}>
  <Card title="创始人决策支持" icon="brain">创始人的思考伙伴。它梳理选项、呈现权衡、起草供复核的推理。它框定决策，不做决策；在有人据此行动之前，其输出仅供参考。</Card>
  <Card title="项目尽调" icon="search">项目尽调的四维度初稿。它从许可来源检索、填充尽调清单，并为每条主张标注来源。数字默认标「待核实」；它不自行联系项目方。</Card>
  <Card title="触达与会议纪要" icon="mail">触达邮件草稿，加会议转录与纪要。它按模板产出行动项、owner 与跟进草稿。任何内容都不自动发送；由节点复核后人工发送。</Card>
  <Card title="节点支持" icon="message-circle">节点问答与 FAQ 自动化。它从已批准材料中作答，并把超出范围的问题路由给人。它不擅自编造政策，也不代表节点发声。</Card>
  <Card title="PoB 证据校验" icon="circle-check">对提交证据的初审。它按必填字段核对证据包，并为证明账本标记缺口。它做预筛与分类；从不最终裁定 PoB 记录。</Card>
</CardGrid>

## 为什么每个 Agent 是独立单元

单一万能 Agent 共享一套工具集，会破坏最小权限原则。把五个分开，系统才能为每一个固定：它可看什么、可做什么、必须记录什么。

<CardGrid cols={2}>
  <Card title="各自可看什么" icon="eye">项目尽调读取广泛材料；节点支持读取已批准的 FAQ 内容；PoB 证据校验只读取提交的证据包与必填字段。</Card>
  <Card title="各自可做什么" icon="wrench">触达与会议纪要起草纪要与邮件；节点支持在已批准材料内作答。没有任何 Agent 可自行转移资金或签署。</Card>
  <Card title="各自必须记录什么" icon="clipboard-list">项目尽调记录其查询与检索片段；触达记录源会议与时间戳；证据校验记录核对了哪些字段、标记了什么。</Card>
  <Card title="各自如何抵达 PoB" icon="file-text">归因模板按 Agent 拆分，因此被采纳的尽调初稿与被采纳的触达纪要带不同的 Proof 字段，在结算时不会冲突。</Card>
</CardGrid>

## 扩展节奏

五个 MVP Agent 是起点，不是上限。白皮书给出分阶段路径：集群在 Phase 1 增至 10 个 Agent，Phase 2 增至 25 个，Phase 3 增至 40 个成熟 Agent。

<Statement>每个新增 Agent 与最初五个以同样方式进入：明确位置、受限权限集、输出日志、归因记录。增长为集群添加单元，不放松边界。</Statement>

## 常驻管理集群

在 MVP Agent 之下，WCN 自身运营由创始人加八个常驻 Agent 运转。一个担任总指挥，六个覆盖法务、资本、网络、知识、品牌、运营六个支柱域，两个为横切体：一个维护底层，一个负责审计。

每个常驻 Agent 各持一条命门（绝不可为之事）与一条恒真账本断言。资本 Agent 绝不动钱；品牌 Agent 未经授权绝不发布。模式与 MVP Agent 相同：固定范围、固定边界、稳得住的记录。

<Callout type="warning">
不做分离，权限会扩张，日志无法辩护，PoB 记录无法计价。Agent 的类型是治理与结算的前提，不是标签。
</Callout>

## 一个事件，多个 Agent

<Steps>
  <Step title="项目尽调">读取公开融资新闻与内部备忘，产出冲突点与待问清单。</Step>
  <Step title="创始人决策支持">梳理该清单中的权衡，为下一轮讨论起草推理，供创始人复核。</Step>
  <Step title="触达与会议纪要">把会议结论写成带 owner 与到期日的行动项，并附到任务上。</Step>
  <Step title="PoB 证据校验">按必填字段核对由此产生的证据包，并在证明账本复核之前标记任何缺口。</Step>
</Steps>

<Statement>分离 Agent 的意义，是在出问题时回答一个问题：是模型、工具、配置，还是人类采纳环节。只有先拆分工作，才能划出边界。</Statement>
