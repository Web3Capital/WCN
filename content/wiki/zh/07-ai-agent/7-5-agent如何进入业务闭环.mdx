## 进入闭环的主路径

<Callout type="info">
任务带类型与验收标准被派发。Agent 在授权内执行。结构化输出落库。节点 owner 采纳、驳回或修订。状态机推进 deal 或任务。结果证据被提交。证明账本复核。生成一条 PoB 记录。该条目进入结算与记录。
</Callout>

<Statement>Agent 的价值，是作为可追溯的结果推动力。系统区分「生成了十页」与「其中三段被采纳，并促成一次会议召开、一项条款修改，或一个付款条件满足」。</Statement>

## Agent 输出抵达 Proof 需要什么

<CardGrid cols={2}>
  <Card title="绑定任务" icon="link">每条输出系于一个任务句柄。游离在私聊或离线文档中的松散结果不具资格。</Card>
  <Card title="留痕输入与版本" icon="clipboard-list">模型名与版本、参数、检索片段，以及关键附件哈希——足以复现输出，并在争议中取证。</Card>
  <Card title="记录人类采纳" icon="circle-check">一个明确事件：谁采纳、何时、有何修订。与隐式的复制粘贴不同，这是结构化且被保留的。</Card>
  <Card title="关联结果" icon="refresh-cw">可追溯到一个 deal、一个里程碑或一个业务事件。没有这条关联，归因与结算无法进行。</Card>
</CardGrid>

<Callout type="warning">
缺任一条件，输出按工具生成物处理，不计入 PoB。理由与会计相同：没有凭据与工作底稿的条目，无法入账。
</Callout>

## 三个具体示例

<Steps>
  <Step title="尽调材料">任务：A 轮材料的风险章节。项目尽调拉取可比风险披露，并带来源起草该章节。节点修订并采纳，再上传修订后的文件哈希。Proof 将其关联到投资人会议所用的定稿，会议结束后随之产生一条 PoB 记录。</Step>
  <Step title="证据复核">一项已完成任务提交证据包。PoB 证据校验按必填字段核对，标记两处缺口。节点补齐缺口，只有补齐的证据包进入证明账本；证据校验的初审不会让它自行通过。</Step>
  <Step title="监控跟进">一份监控说明报告异常。触达与会议纪要起草一条内部任务说明。风险负责人采纳其中一条并创建跟进任务。回执进入证据包，Proof 通过后该条目进入 PoB。</Step>
</Steps>

## 反例：看似很忙，没有闭环

- 一次开放、无界的运行产出大量笔记，没有任务、没有 owner。那是噪音。
- 一个 Agent 起草了一封漂亮的邮件，但不从 WCN 任务发送，也不带采纳标记。那无法归因。
- 一个 Agent 提出一步建议，节点口头转发，系统不持有任何状态。那无法验证。

## 这一步决定什么

<CardGrid cols={2}>
  <Card title="工具，还是执行层" icon="bot">Agent 工作只有在进入闭环后，才成为网络的执行层。在那之前，它与任何模型封装无异。</Card>
  <Card title="可结算，还是不可" icon="coins">没有 Proof 链就没有 PoB，激励与分润便没有可依的证据。</Card>
  <Card title="可治理，还是不可" icon="shield">采纳率、被回滚的采纳与模型对比都可衡量，因此一套 Agent 配置可依据证据升级或暂停。</Card>
  <Card title="WCN 的差异" icon="zap">关键不在于它接入了一个模型。而在于机器输出，在多节点业务网络中，抵达同一套证据与结算语言。</Card>
</CardGrid>

<Callout type="tip">
一个验收测试：在争议中，日志能否在半小时内说清谁让 Agent 做了什么、采纳了什么、最终业务结果是什么。若能，闭环就是稳的。
</Callout>
