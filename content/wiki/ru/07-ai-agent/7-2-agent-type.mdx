## Пять типичных агентов

<CardGrid cols={2}>
  <Card title="Исследовательский агент" icon="search">Поиск информации, сводка из нескольких источников, структурированная таблица сопоставимых компаний/треков и заполнение списка комплексной проверки. **Примеры инструментов**: RAG для документов интрасети, разрешенный API сторонних данных, запрещающий прямое подключение к любому сканированию веб-страниц (защита от внедрения). **Примечание LLM**. Ссылки на факты должны иметь привязки к источнику; значения по умолчанию помечены как «для проверки».</Card>
  <Card title="Агент по сделке" icon="handshake">Кандидаты на соответствие капитальным проектам, питч-драфты на уровне тизеров, рекомендации по статусу сделки. **Запрещено**: предоставление обязательных условий или эксклюзивных обязательств от имени узлов без разрешения. **Сравнение второго пилота**: второй пилот оптимизирует рукописи, написанные одним человеком; Выходные данные агента по сделкам должны содержать Deal_id и сводку логики рекомендаций для проверки.</Card>
  <Card title="Агент роста" icon="trending-up">Копия с несколькими версиями, маркированные списки воронки, сводка атрибуции после кампании (связанная с задачей роста). **Риск**: Перед публикацией во внешнем мире необходимо пройти через шлюз ручного выпуска; избегайте сочетания с наводящими словами, такими как «обязательство о прибыли».</Card>
  <Card title="Агент исполнения" icon="settings">Протоколы встреч, разбивка дел, черновик напоминаний и список отслеживания квитанций. **Эталон**: наиболее близок к корпоративной автоматизации (Zapier/Make) + LLM, но каждое предложение должно быть связано с заметками о встречах и временными метками.</Card>
  <Card title="Агент ликвидности" icon="droplet">Сводка мониторинга ликвидности и состояния рынка, состояние контрольного списка, описание сигналов тревоги аномального порога. **TradFi Lenovo**: Подобно «слою естественного языка» определенного типа панели управления рисками в Aladdin, право принятия решений по-прежнему принадлежит торговому отделу/узлу.</Card>
</CardGrid>

## Почему их нельзя смешать в «суперагента»?

Агенты общего назначения в стиле LangChain часто используют один и тот же набор инструментов; в институциональных сценариях **принцип наименьших привилегий** будет нарушен. После классификации система может выполнять:

<CardGrid cols={2}>
  <Card title="Кто что может увидеть" icon="eye">Исследования позволяют прочитать широкий спектр материалов; Ликвидность может определять только рыночные цены и согласованные контракты; По умолчанию сделка не считывает конфиденциальные поля уровней зарплат во всей базе данных.</Card>
  <Card title="Кто что может сделать" icon="wrench">Исполнение может писать задания и комментарии; Рост может писать только маркетинговые проекты областей; ни один тип Агента не может переводить деньги или подписывать контракты по умолчанию.</Card>
  <Card title="Кто какой журнал должен вести?" icon="clipboard-list">Исследование записывает поисковый запрос и идентификатор фрагмента; Deal записывает рекомендуемые функции и версии; Ликвидность фиксирует определение индикатора и время получения данных.</Card>
  <Card title="Как и кто попадает в PoB" icon="circle-check">Шаблоны атрибуции разделены по типам: например, «Принятые абзацы исследований» и «Принятые материалы роста» имеют разные поля «Доказательства», чтобы избежать споров по урегулированию.</Card>
</CardGrid>

## Отображение отношений с помощью многоагентной оркестрации

*Агент/узел* в CrewAI/LangGraph можно сравнить с типами WCN, но WCN исправляет три дополнительных вещи: **(1)** Белый список инструментов, разрешенный для каждого типа; **(2)** The state machine with the human approval edge; **(3)** The output is bound to `task_id`, node adoption event. Without (3), no matter how beautiful the arrangement is, it is just an internal experiment.
