# PRD-04: Project & Asset System

> Status: Draft | Priority: P0 — Core | Owner: Product Lead
> Dependencies: PRD-01 (Identity), PRD-02 (Node Management)
> Affects: PRD-05 (Capital), PRD-06 (Deal Room), PRD-07 (Task)

---

## 1. Overview

The Project & Asset System is Layer 1 of WCN's five-layer architecture — the demand entry point. It manages how projects and assets enter the network, how their needs are structured, and how they become matchable for capital, services, and distribution. Without real projects and assets in the system, all subsequent layers are idle.

**Core Principle**: A project in WCN is not a pitch deck — it is a structured demand signal with specific needs (capital, services, growth), current status, and matchable attributes. The system must transform unstructured project information into structured, actionable data.

## 3. Core User Stories

### Project Intake
- US-4.1: As a project node, I can submit my project with structured information: name, sector, stage, team, product status, and specific needs.
- US-4.2: As a project node, I can specify my funding needs: target amount, valuation range, instruments (SAFE/SAFT/Token Warrant/Equity), and timeline.
- US-4.3: As a project node, I can specify service needs: legal, audit, security, development, growth, marketing.
- US-4.4: As a project node, I can upload materials: Deck, Tokenomics, technical docs, audit reports.

### Project Discovery
- US-4.5: As a capital node, I can browse and search projects filtered by sector, stage, funding size, and region.
- US-4.6: As a service node, I can discover projects that need my specific services.
- US-4.7: As a system, I provide structured project summaries (auto-generated by Research Agent) for quick evaluation.

### Project Management
- US-4.8: As a project node, I can track my project status: intake → matching → in-deal → funded → post-deal.
- US-4.9: As a project node, I can update my information as my project evolves (new round, new needs, status changes).
- US-4.10: As a platform admin, I can flag, archive, or remove projects that violate quality standards.

### Asset Registry (Phase 2)
- US-4.11: As an RWA project, I can register tokenized assets with structured metadata.
- US-4.12: As a system, I maintain a registry of verified assets available for distribution and settlement.

## 4. Data Model

```
Project
├── id: string
├── nodeId: string (FK → Node, the project node)
├── name: string
├── slug: string (unique, URL-friendly)
├── sector: enum (DEFI, AI, RWA, INFRA, GAMING, SOCIAL, PAYMENTS, DEPIN, OTHER)
├── stage: enum (IDEA, MVP, TESTNET, MAINNET, GROWTH, MATURE)
├── status: enum (INTAKE, MATCHING, IN_DEAL, FUNDED, POST_DEAL, ARCHIVED)
├── description: string
├── website: string?
├── logo: string?
├── teamSize: int?
├── foundedDate: string?
├── chains: string[] (e.g., ["Ethereum", "Base", "Solana"])
├── tokenStatus: enum (NO_TOKEN, PLANNING, LIVE)
├── tokenTicker: string?
├── createdAt: datetime
└── updatedAt: datetime

FundingNeed
├── id: string
├── projectId: string (FK → Project)
├── roundName: string ("Seed", "Pre-A", "Strategic", etc.)
├── targetAmount: decimal
├── raisedAmount: decimal
├── valuationRange: string? ("$10M-$15M pre-money")
├── instruments: string[] (["SAFE", "Token Warrant"])
├── minTicket: decimal?
├── maxTicket: decimal?
├── timeline: string? ("Q2 2026")
├── status: enum (OPEN, PARTIALLY_FILLED, CLOSED)
├── investorPreferences: string? (sector expertise, strategic value, etc.)
└── createdAt: datetime

ServiceNeed
├── id: string
├── projectId: string (FK → Project)
├── serviceType: enum (LEGAL, AUDIT, SECURITY, DEVELOPMENT, GROWTH, MARKETING, RESEARCH, TOKENOMICS, OTHER)
├── description: string
├── urgency: enum (LOW, MEDIUM, HIGH, CRITICAL)
├── budget: string?
├── status: enum (OPEN, MATCHED, IN_PROGRESS, COMPLETED)
└── createdAt: datetime

ProjectMaterial
├── id: string
├── projectId: string (FK → Project)
├── type: enum (DECK, TOKENOMICS, TECHNICAL_DOC, AUDIT_REPORT, LEGAL_OPINION, FINANCIAL_MODEL, OTHER)
├── fileName: string
├── fileUrl: string
├── fileSize: int
├── uploadedBy: string (FK → User)
├── isConfidential: boolean
└── createdAt: datetime

ProjectSummary (AI-generated)
├── id: string
├── projectId: string (FK → Project)
├── version: int
├── executiveSummary: string
├── strengths: string[]
├── risks: string[]
├── competitorAnalysis: string?
├── generatedBy: string ("research-agent-v1")
└── createdAt: datetime
```

## 6. Key Flows

### Project Intake Flow
```
Project node opens /dashboard/projects → "Add Project"
  → Step 1: Basic info (name, sector, stage, description, chains)
  → Step 2: Funding needs (round, amount, valuation, instruments)
  → Step 3: Service needs (what services needed, urgency)
  → Step 4: Upload materials (deck, docs)
  → Submit → Project created (INTAKE status)
  → Research Agent auto-generates summary (async)
  → Admin review (optional for quality) → Status: MATCHING
  → Capital/service nodes discover via search/matching
```

### Project → Deal Flow
```
Capital node discovers project → Expresses interest
  → System creates Deal Room (PRD-06)
  → Project status changes to IN_DEAL
  → ... (Deal Room flow)
  → Deal closes → Project status: FUNDED
```

## 8. UI Requirements

### Project List (`/dashboard/projects`)
- Card or table view with: name, sector badge, stage, funding status, last updated
- Filters: sector, stage, funding status, region
- Search by name or description
- Quick actions: view, edit (own), archive

### Project Detail (`/dashboard/projects/:id`)
- Overview tab: basic info, description, chains, team
- Funding tab: current round details, progress bar, investor preferences
- Services tab: list of service needs with status
- Materials tab: uploaded documents with download links
- Activity tab: timeline of events (status changes, deals, matches)
- AI Summary tab: auto-generated research summary

### Project Intake Form (`/dashboard/projects/new`)
- Multi-step wizard with validation
- Auto-save draft support
- Material upload with drag-and-drop
- Preview before submit
